# Life Project 下一步开发计划

## 1. 本轮目标

在现有 `V1` 的单日任务能力之上，补上更适合“习惯管理 / 周期打卡”的能力，让产品从“今日任务页”升级为“任务 + 周期习惯 + 数据导入 + 热力图统计”的组合型 Web App。

这一轮的核心不是继续扩展提醒、协作、外部日历集成，而是先把下面三件事打通：

1. 每日 / 每周 / 自定义周期任务
2. 按日期展示的热力图
3. 导入表与导入流程，支持从 `resource` 里的样例数据迁移

## 2. 这一步要新增的能力

### 2.1 周期任务

把当前一次性任务和“重复出现的任务”分开管理。

建议支持的周期类型：

- 每日
- 每周
- 自定义频率

建议支持的编辑字段：

- 标题
- 描述
- 周期类型
- 每周重复的星期几
- 开始日期
- 结束日期
- 优先级
- 分类
- 颜色或主题标识
- 归档状态

建议保留两层数据：

- `tasks` 继续保存一次性任务
- 新增一组“周期任务模板 + 每日记录”表，用来承载重复任务和打卡历史

这样做的好处是：

- 不会破坏现有 `/today` 的单日任务逻辑
- 周期任务可以单独维护规则
- 热力图可以直接从历史记录汇总

### 2.2 热力图

增加一个基于日期的热力图页面或卡片区域，用来展示用户的活跃度。

热力图建议支持：

- 最近 12 个月
- 按天显示颜色强度
- 悬浮查看当天明细
- 切换统计口径

建议的统计口径：

- 完成任务数
- 周期任务打卡数
- 综合活跃度

如果后面要做得更细，可以再拆成：

- 任务热力图
- 习惯热力图
- 综合热力图

但这一轮先做一个统一热力图就够了。

### 2.3 导入表

增加一个“导入表”流程，而不是直接把文件一股脑塞进数据库。

导入流程建议分成 4 步：

1. 上传文件
2. 预览表格
3. 字段映射
4. 确认导入

导入表需要显示的内容：

- 原始列名
- 目标字段
- 示例值
- 校验状态
- 错误信息

建议支持的输入：

- 单个 `CSV`
- `ZIP` 压缩包
- 预览后再确认写入

## 3. 资源导入样例

当前 `resource` 里已经有一个很适合做导入样例的数据包：

- `resource/Loop Habits CSV 2026-07-05.zip`

从文件结构看，它包含：

- `Habits.csv`
- `Scores.csv`
- `Checkmarks.csv`
- 按习惯拆分的子目录数据，例如 `001/`、`002/`、`006 codeforces/`

这份样例非常适合用来验证导入流程，因为它同时覆盖了：

- 周期任务模板
- 历史数值
- 打卡状态
- 多文件组合导入

## 4. 样例数据映射建议

### 4.1 `Habits.csv`

这一份更像“周期任务模板表”。

建议映射关系：

- `Name` -> 任务 / 习惯标题
- `Type` -> 任务类型
- `Question` -> 每日提示文案
- `Description` -> 说明
- `FrequencyNumerator` / `FrequencyDenominator` -> 周期规则
- `Color` -> 颜色标识
- `Unit` -> 统计单位
- `Target Type` -> 目标类型
- `Target Value` -> 目标值
- `Archived?` -> 归档状态

### 4.2 `Checkmarks.csv`

这一份更像“每日完成记录表”。

建议映射关系：

- `Date` -> 记录日期
- 每一列的习惯名 -> 对应模板的每日状态

建议处理方式：

- `YES_MANUAL` -> 已完成
- `UNKNOWN` -> 未记录 / 未完成

### 4.3 `Scores.csv`

这一份更像“每日数值记录表”。

建议映射关系：

- `Date` -> 记录日期
- 每一列的习惯名 -> 对应模板的分值

这一份可以直接喂给热力图和统计页，用来做历史活跃度展示。

## 5. 建议的数据结构

这一轮建议不要强行把所有东西塞回 `tasks`，而是拆清楚三层：

### 5.1 一次性任务

继续使用现有 `tasks` 表，承载今天要做、某一天才做的独立任务。

### 5.2 周期任务模板

用于描述“每天都要做”“每周一三五做”“需要长期追踪”的内容。

建议字段：

- `id`
- `user_id`
- `title`
- `description`
- `task_type`
- `frequency_rule`
- `question`
- `unit`
- `target_type`
- `target_value`
- `color`
- `archived_at`
- `created_at`
- `updated_at`

### 5.3 周期任务记录

用于保存某天是否完成、完成了多少、备注是什么。

建议字段：

- `id`
- `template_id`
- `user_id`
- `record_date`
- `value`
- `is_done`
- `notes`
- `source`
- `created_at`
- `updated_at`

### 5.4 导入作业表

导入最好有独立记录，方便回溯和重试。

建议字段：

- `id`
- `user_id`
- `source_name`
- `source_type`
- `status`
- `total_rows`
- `success_rows`
- `failed_rows`
- `created_at`
- `finished_at`

### 5.5 导入明细表

用于存每一行的解析结果和错误原因。

建议字段：

- `id`
- `job_id`
- `sheet_name`
- `source_key`
- `raw_payload`
- `mapped_payload`
- `status`
- `error_message`

## 6. 页面和交互建议

建议拆成四个入口：

1. `/today`
   - 继续作为今日任务主页面
   - 展示当天的一次性任务和周期任务概览
2. `/habits`
   - 周期任务管理页
   - 新建、编辑、归档、查看历史
3. `/heatmap`
   - 热力图统计页
   - 可切换统计维度
4. `/import`
   - 导入表页
   - 上传、预览、映射、确认导入

如果后面想更轻，也可以把这四个能力做成同一套 Tab，而不是四个完全独立页面。

## 7. 推荐开发顺序

### 第 1 步：数据模型

- 先补周期任务模板表
- 再补周期任务记录表
- 再补导入作业和导入明细表

### 第 2 步：导入表

- 先支持单个 CSV 预览
- 再支持 ZIP 组合包
- 最后接入 `Loop Habits` 样例

### 第 3 步：周期任务页面

- 新建 / 编辑 / 归档
- 周期规则配置
- 每日打卡

### 第 4 步：热力图

- 先出基础日历格子

### 第 5 步：联调和收口

- 检查时区和日期边界
- 防止导入重复
- 保持 `/today` 现有体验不退化

## 8. 这轮先不做的事

- 不做协作共享
- 不做提醒推送
- 不做外部日历同步
- 不做复杂标签体系
- 不做高级报表

## 9. 这一轮的验收标准

满足下面几条，就可以认为这一步完成：

1. 用户可以创建周期任务
2. 用户可以按日期完成打卡
3. 用户可以看到至少一个热力图视图
4. 用户可以导入 `resource/Loop Habits CSV 2026-07-05.zip`

6. 现有 `/today` 页面依然可用

## 10. 导出功能

导出功能建议和导入功能配套实现，目标是让用户既能把本地数据导进去，也能把当前账号里的数据安全导出来做备份或迁移。

### 10.1 导出范围

建议先支持下面几类导出：

- 当前日期范围内的一次性任务
- 周期任务模板
- 周期任务记录
- 热力图统计数据
- 完整备份包

### 10.2 导出格式

建议优先支持：

- `CSV`，适合表格工具和二次处理
- `ZIP`，适合把多个表一起打包导出
- `JSON`，适合程序化备份和恢复

### 10.3 导出入口

导出入口可以放在两个地方：

- 独立 `/export` 页面
- `/import` 页面里的“导入 / 导出”切换区

如果想减少页面数量，建议先做后者，后面再拆分成独立页面。

### 10.4 导出与导入的配套关系

理想状态下，导出和导入应该尽量互相兼容：

- 从导出包里拿出来的数据，应该能再次导回系统
- 对于 `Loop Habits` 这类样例，最好能做到“导入一遍后再导出，结构基本一致”
- 导出时尽量保留原始字段和映射字段，方便排查问题

### 10.5 导出功能的验收标准

这一项可以作为下一阶段的补充验收：

1. 用户可以按日期范围导出任务
2. 用户可以导出周期任务和打卡记录
3. 用户可以把导出的文件重新导入回来
4. 导出结果可用于备份和迁移

## 11. 下一轮体验与数据修正计划

这一轮的目标是把已经存在的功能从“能用”推进到“符合日常长期使用”。重点不再是继续堆新入口，而是修正中文、本地化、移动端、导入准确性、完成度、分数和单习惯详情。

### 11.1 中文与响应式适配

- 所有页面、导航、按钮、提示、状态、表头、空状态、错误文案均使用中文。
- `/today`、`/habits`、`/heatmap`、`/import`、`/export`、登录页和注册页都必须同时适配 PC 和手机端。
- 手机端优先保证核心操作可完成：切换日期、录入习惯、查看热力图、导入预览、导出备份。
- 导航在手机端需要更紧凑，避免按钮挤压、换行错乱或文字溢出。

### 11.2 导入准确性修正

- 修复 `Loop Habits` 导入中 `睡觉`、`起床` 这类 `YES_NO` 习惯被导入为小数的问题。
- 对 `YES_NO` 类型，导入后每日值只能归一化为 `0` 或 `1`。
- `Checkmarks.csv` 优先作为完成状态来源，`Scores.csv` 只能作为数值或分数参考，不能覆盖 `YES_NO` 的二值语义。
- 没有数据的日期自动视为 `0`，不能视为未知完成。
- 导入后应能清楚看到原始值、归一化值、完成度和完成状态。

### 11.3 完成判断规则

每个习惯需要根据目标类型自动判断某天是否完成：

- `AT_LEAST`：实际值大于或等于目标值，视为完成。
- `AT_MOST`：实际值小于或等于目标值，视为完成。
- `YES_NO`：值为 `1` 或明确完成状态，视为完成。
- 没有数据：实际值按 `0` 处理。
- 用户仍可手动修正某天记录，但系统默认判断必须稳定一致。

### 11.4 每日完成度规则

每个习惯每天都需要一个 `completion_ratio`，范围为 `0` 到 `1`。

- 彻底完成目标为 `1`。
- 完全没有完成为 `0`。
- `AT_LEAST` 数值型目标：`completion_ratio = min(actual / target, 1)`。
- `AT_MOST` 数值型目标：小于或等于目标值为 `1`；达到 `2 * target` 时为 `0`；中间按线性比例下降。
- `YES_NO`：完成为 `1`，未完成或无数据为 `0`。
- 计数最多保留三位小数。
- 导入或展示时要处理三位小数缩放场景，例如 `7000` 应被视为 `7`。

### 11.5 分数表

每个习惯需要增加独立分数表或等价的数据结构，用于保存基于每日完成度计算出的分数。

- 每个习惯每天有一个分数，范围 `0` 到 `100`。
- 分数受每日完成度影响。
- 越近的日期权重越高，越久远的日期影响越弱。
- 满分为 `100`，最低分为 `0`。
- 分数应可重新计算，避免导入修正后历史分数无法更新。
- 热力图、习惯详情页、导出数据都应优先使用统一计算结果。

### 11.6 日期与北京时间

- 所有业务日期统一按北京时间 `Asia/Shanghai` 计算。
- 今日、日期切换、导入日期、导出日期、热力图范围都必须使用北京时间语义。
- 避免因为用户设备本地时区不同，导致日期前后偏移。
- 如需要当前日期，应优先封装统一的北京时间日期工具，而不是在页面里直接散落使用 `new Date()`。

### 11.7 日历型日期修改

- 当前简单的前一天 / 后一天 / 日期输入需要升级为更方便的日历型修改。
- 至少在 `/today` 和 `/habits` 中提供清晰的日期选择体验。
- 手机端应能快速切换今天、昨天、明天和选择具体日期。
- 后续可扩展为周视图或月视图，但本轮先完成可靠的日期选择组件。

### 11.8 习惯删除

- 习惯应该可以删除。
- 删除前需要确认，避免误删。
- 优先使用软删除或归档 + 删除确认策略，确保历史记录不会被意外破坏。
- 如果执行真正删除，需要同步处理该习惯的每日记录、分数记录和导入映射。

### 11.9 单个习惯详情页

每个习惯都应该有独立页面，例如 `/habits/[id]`。

详情页应包含：

- 习惯基础信息
- 该习惯自己的热力图
- 每日完成度趋势
- 分数趋势
- 编辑、归档、删除入口

### 11.10 下一轮验收标准

完成后至少满足：

1. 所有按钮和主要界面文字均为中文。
2. 所有主要页面在 PC 和手机端布局正常。
3. 导入 `resource/Loop Habits CSV 2026-07-05.zip` 后，`睡觉`、`起床` 不再出现错误小数。
4. 每个习惯每天都有明确的实际值、完成状态、完成度和分数。
5. 无数据日期自动按 `0` 参与计算。
6. 日期计算统一按北京时间。
7. 可以用日历方式切换日期。
8. 习惯可以删除。
9. 每个习惯有独立详情页。
10. 热力图和分数表使用新的完成度规则。
11. 构建、类型检查和核心导入样例验证通过。
